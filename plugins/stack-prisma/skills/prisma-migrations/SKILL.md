# Prisma Migrations

마이그레이션 생성, 관리, 롤백 전략입니다.

## Migration Commands

```bash
# Development
npx prisma migrate dev --name <migration_name>  # Create & apply
npx prisma migrate dev --create-only            # Create only, don't apply

# Production
npx prisma migrate deploy                        # Apply pending migrations
npx prisma migrate status                        # Check migration status

# Reset (DANGER - deletes all data)
npx prisma migrate reset                         # Reset DB and apply all

# Resolve issues
npx prisma migrate resolve --applied <name>      # Mark as applied
npx prisma migrate resolve --rolled-back <name>  # Mark as rolled back
```

## Naming Conventions

```yaml
format: YYYYMMDDHHMMSS_descriptive_name

examples:
  - 20240115120000_init
  - 20240115130000_add_user_profile
  - 20240115140000_add_post_categories
  - 20240116100000_make_email_unique
  - 20240116110000_add_soft_delete_to_posts

naming_patterns:
  create_table: add_<table_name>
  add_column: add_<column>_to_<table>
  remove_column: remove_<column>_from_<table>
  add_index: add_index_on_<table>_<columns>
  add_relation: add_<relation>_to_<table>
  modify_column: change_<column>_in_<table>
```

## Safe Migration Patterns

### Adding a New Table (Safe)

```prisma
// Before: nothing

// After:
model Category {
  id        String   @id @default(uuid())
  name      String
  slug      String   @unique

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@map("categories")
}
```

### Adding Nullable Column (Safe)

```prisma
// Before:
model User {
  id    String @id
  email String
}

// After:
model User {
  id    String  @id
  email String
  bio   String? // New nullable column - SAFE
}
```

### Adding Required Column with Default (Safe)

```prisma
// Before:
model User {
  id    String @id
  email String
}

// After:
model User {
  id       String   @id
  email    String
  isActive Boolean  @default(true) // Has default - SAFE
}
```

## Dangerous Migration Patterns

### Adding Required Column Without Default (DANGEROUS)

```yaml
problem: |
  Adding non-nullable column without default fails if table has data

wrong:
  model User {
    bio String  // Will fail!
  }

solutions:
  1_add_with_default: |
    model User {
      bio String @default("")
    }
    # Then remove default in next migration if needed

  2_multi_step: |
    Step 1: Add nullable
      bio String?
    Step 2: Backfill data
      UPDATE users SET bio = '' WHERE bio IS NULL
    Step 3: Make required
      bio String
```

### Renaming Column (DANGEROUS)

```yaml
problem: |
  Prisma treats rename as drop + create = DATA LOSS

wrong:
  # Just renaming in schema causes data loss

safe_approach: |
  Step 1: Add new column
    model User {
      firstName String?  // Old
      givenName String?  // New
    }

  Step 2: Copy data (custom SQL migration)
    UPDATE users SET givenName = firstName;

  Step 3: Remove old column
    model User {
      givenName String?
    }
```

### Changing Column Type (DANGEROUS)

```yaml
problem: |
  Type changes may fail or lose data

examples:
  string_to_int: |
    # Fails if non-numeric data exists
    # Manually cast: ALTER TABLE ... USING column::integer

  int_to_string: |
    # Usually safe, but verify

  reducing_varchar: |
    # May truncate data
    # Check: SELECT MAX(LENGTH(column)) FROM table

safe_approach: |
  1. Check existing data compatibility
  2. Create custom migration with explicit CAST
  3. Test on staging first
```

## Custom SQL Migrations

### Adding Custom SQL

```sql
-- migration.sql (in migrations folder)

-- Custom data migration
UPDATE users SET role = 'USER' WHERE role IS NULL;

-- Add check constraint
ALTER TABLE orders ADD CONSTRAINT check_positive_amount
  CHECK (amount > 0);

-- Create partial index
CREATE INDEX idx_active_users ON users (email)
  WHERE deleted_at IS NULL;
```

### Backfill Pattern

```sql
-- 20240115_add_slug_to_categories/migration.sql

-- Step 1: Add nullable column (auto-generated by Prisma)
ALTER TABLE categories ADD COLUMN slug VARCHAR(255);

-- Step 2: Backfill data (manual)
UPDATE categories
SET slug = LOWER(REPLACE(name, ' ', '-'))
WHERE slug IS NULL;

-- Step 3: Make non-nullable (manual or next migration)
ALTER TABLE categories ALTER COLUMN slug SET NOT NULL;

-- Step 4: Add unique constraint
ALTER TABLE categories ADD CONSTRAINT categories_slug_unique UNIQUE (slug);
```

## Rollback Strategies

### Manual Rollback Script

```sql
-- rollback_20240115_add_user_profile.sql

-- Reverse the migration
DROP TABLE IF EXISTS user_profiles;

ALTER TABLE users DROP COLUMN IF EXISTS profile_id;
```

### Zero-Downtime Migration

```yaml
pattern: expand_contract

expand_phase:
  - Add new column (nullable)
  - Deploy new code that writes to both columns
  - Backfill old data to new column

contract_phase:
  - Deploy code that reads from new column only
  - Remove old column
  - Make new column required if needed

example:
  # Renaming email to emailAddress
  expand:
    1. Add emailAddress String?
    2. Deploy: write to both email and emailAddress
    3. Backfill: UPDATE users SET emailAddress = email

  contract:
    4. Deploy: read from emailAddress only
    5. Remove email column
    6. Make emailAddress required
```

## Migration Checklist

```yaml
before_migration:
  - [ ] Backup database
  - [ ] Review generated SQL
  - [ ] Check for data loss risks
  - [ ] Test on staging environment
  - [ ] Plan rollback strategy

after_migration:
  - [ ] Verify migration applied
  - [ ] Check application functionality
  - [ ] Monitor for errors
  - [ ] Run prisma generate

production_checklist:
  - [ ] Schedule during low-traffic period
  - [ ] Notify team
  - [ ] Have rollback script ready
  - [ ] Monitor database performance
```

## Common Issues

```yaml
migration_drift:
  symptom: "Drift detected" error
  cause: Manual DB changes not in migrations
  fix: |
    npx prisma db pull  # Pull DB state to schema
    # Or reset and reapply migrations

shadow_database_error:
  symptom: Cannot create shadow database
  cause: No permission or connection issues
  fix: |
    # Use different shadow DB URL
    shadowDatabaseUrl = env("SHADOW_DATABASE_URL")

migration_history_conflict:
  symptom: Migration already applied
  fix: |
    npx prisma migrate resolve --applied <name>
```
