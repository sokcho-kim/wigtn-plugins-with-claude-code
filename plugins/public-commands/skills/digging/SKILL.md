# Digging

PRD 문서의 취약점, 누락점, 리스크를 분석하여 구현 전 계획을 개선합니다.

## Pipeline Position

```
┌─────────────────────────────────────────────────────────────┐
│  [/prd] → [digging] → [/implement] → [/auto-commit]        │
│            ^^^^^^^^                                         │
│            현재 단계                                         │
└─────────────────────────────────────────────────────────────┘
```

| 이전 단계 | 현재 | 다음 단계 |
|----------|------|----------|
| `/prd` - PRD 문서 생성 | `digging` - PRD 분석 & 개선 | `/implement` - 구현 시작 |

## When to Use This Skill

### 자동 실행 (필수)
- **`/prd` 명령 완료 후**: 자동으로 digging이 실행되어 PRD 품질 검증
- **`/implement` 시작 시**: 품질 게이트 확인 (Critical 이슈 있으면 블로킹)

### 수동 실행 (선택)
- 기존 PRD 문서의 품질을 높이고 싶을 때
- PRD 수정 후 재검증이 필요할 때
- 프로젝트 리스크를 사전에 파악하고 싶을 때

## Quality Gate Integration

digging 분석 결과는 `/implement` 실행 시 품질 게이트로 사용됩니다:

```
┌─────────────────────────────────────────────────────────────┐
│  품질 게이트 판단 기준                                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Critical 이슈 0개 → ✅ PASS → /implement 진행 가능          │
│  Critical 이슈 1개+ → ❌ BLOCKED → 수정 필요                 │
│                                                             │
│  Major/Minor 이슈는 구현 중 해결 가능                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Critical 이슈 기준:**
- 보안 취약점 (Rate Limiting 미정의, 인증 정책 누락 등)
- 핵심 기능 누락 (필수 요구사항 미정의)
- 구현 불가능한 요구사항
- 데이터 무결성 위험

## Analysis Categories

### 1. 요구사항 완전성 (Completeness)

| 체크 항목 | 설명 |
|----------|------|
| 기능 요구사항 누락 | 명시되지 않은 필수 기능이 있는가? |
| 비기능 요구사항 누락 | 성능, 보안, 확장성 요구가 정의되었는가? |
| 엣지 케이스 | 예외 상황이 고려되었는가? |
| 에러 처리 | 실패 시나리오가 정의되었는가? |
| 사용자 시나리오 | 모든 사용자 유형이 고려되었는가? |

### 2. 기술적 실현 가능성 (Feasibility)

| 체크 항목 | 설명 |
|----------|------|
| 기술 스택 적합성 | 선택한 기술이 요구사항에 적합한가? |
| 구현 복잡도 | 과도하게 복잡한 요구사항이 있는가? |
| 의존성 리스크 | 외부 서비스/라이브러리 의존도가 높은가? |
| 성능 병목 | 예상되는 성능 이슈가 있는가? |
| 확장성 한계 | 스케일링 시 문제가 예상되는가? |

### 3. 보안 및 리스크 (Security & Risk)

| 체크 항목 | 설명 |
|----------|------|
| 인증/인가 | 접근 제어가 명확하게 정의되었는가? |
| 데이터 보호 | 민감 데이터 처리 방안이 있는가? |
| 입력 검증 | 사용자 입력 검증이 고려되었는가? |
| API 보안 | Rate limiting, CORS 등이 정의되었는가? |
| 규정 준수 | GDPR, 개인정보보호법 등 고려되었는가? |

### 4. 일관성 및 명확성 (Consistency)

| 체크 항목 | 설명 |
|----------|------|
| 용어 일관성 | 동일 개념에 다른 용어를 사용하는가? |
| 요구사항 충돌 | 상충되는 요구사항이 있는가? |
| 우선순위 명확성 | P0/P1/P2가 적절히 분배되었는가? |
| 의존성 순환 | 순환 의존하는 요구사항이 있는가? |
| 측정 가능성 | 성공 기준이 측정 가능한가? |

## Parallel Analysis Mode

> **Agent Teams 병렬 분석**: 4개 카테고리를 독립 에이전트로 병렬 실행하여 **4x 속도 향상**을 달성합니다.

### 병렬 모드 활성화 조건

| 조건 | 모드 | 이유 |
|------|------|------|
| PRD 섹션 3개 이상 + 문서 500자 이상 | **병렬** (기본) | 충분한 분석 대상 |
| PRD 섹션 3개 미만 또는 500자 미만 | 순차 | 병렬화 오버헤드 > 이득 |
| `--sequential` 플래그 | 순차 | 사용자 명시 |

**`parallel-digging-coordinator` 연동:**

1. PRD 문서 로드 후 `parallel-digging-coordinator`에 위임
2. Coordinator가 4개 에이전트에 각 카테고리 할당:
   - **Agent A**: Completeness (FR/NFR 커버리지, 에지 케이스)
   - **Agent B**: Feasibility (기술 스택 적합성, 구현 복잡도)
   - **Agent C**: Security & Risk (OWASP, 인증/인가, 데이터 보호)
   - **Agent D**: Consistency (용어 일관성, 우선순위, 의존성 순환)
3. 4개 에이전트 완전 독립 병렬 실행 (상태 공유 없음)
4. 결과 병합:
   - 동일 섹션의 동일 이슈 → 중복 제거 (높은 severity 채택)
   - Severity별 정렬 (Critical → Major → Minor)
   - 카테고리별 이슈 수 집계
5. Quality Gate 판정: Critical 0개 = PASS, 1개+ = BLOCKED

```

## Analysis Protocol (Sequential)

### Phase 1: PRD 문서 로드

```bash
# PRD 파일 위치 탐색
Glob: "**/prd/**/*.md"
Glob: "**/docs/prd/**/*.md"
Glob: "**/*-prd.md"
Glob: "**/*-requirements.md"

# PRD 파일 읽기
Read: <found-prd-file>
```

### Phase 2: 체계적 분석

**분석 순서:**
```
1. 전체 구조 파악
   └── 섹션 누락 여부 확인

2. 요구사항 완전성 검토
   └── 각 FR/NFR 항목 분석

3. 기술적 실현 가능성 평가
   └── 구현 난이도 및 리스크

4. 보안 취약점 식별
   └── 잠재적 보안 이슈

5. 일관성 검증
   └── 충돌/모순 발견
```

### Phase 3: 개선안 도출

각 발견사항에 대해:
- **문제점**: 구체적으로 무엇이 문제인가?
- **영향도**: 구현 시 어떤 문제가 발생하는가?
- **개선안**: 어떻게 수정해야 하는가?
- **우선순위**: 얼마나 급하게 수정해야 하는가?

## 상세 분석

### 🔴 Critical (즉시 수정 필요)

#### C-1. 비밀번호 정책 미정의
- **위치**: Section 3.1 - 회원가입 요구사항
- **문제**: 비밀번호 복잡도, 길이, 만료 정책이 없음
- **영향**: 보안 취약점 발생, 규정 미준수 가능
- **개선안**:
  ```markdown
  ### 3.1.2 비밀번호 정책
  - 최소 길이: 8자 이상
  - 복잡도: 대소문자 + 숫자 + 특수문자 1개 이상
  - 만료: 90일마다 변경 권장
  - 재사용: 최근 5개 비밀번호 재사용 금지
  ```

#### C-2. Rate Limiting 미정의
- **위치**: Section 5.1 - API Endpoints
- **문제**: 로그인 API에 rate limiting 없음
- **영향**: Brute force 공격에 취약
- **개선안**: IP당 분당 10회 제한, 5회 실패 시 15분 잠금

### 🟡 Major (구현 전 수정 권장)

#### M-1. 세션 만료 정책 불명확
- **위치**: Section 4.2 - 세션 관리
- **문제**: "적절한 시간 후 만료" - 구체적 시간 없음
- **개선안**: 활성 세션 30분, Remember Me 7일 명시

### 🟢 Minor (개선 제안)

#### m-1. 용어 불일치
- **위치**: 전체
- **문제**: "사용자", "유저", "회원"이 혼용됨
- **개선안**: "사용자"로 통일

## Severity Levels

| Level | 기준 | 액션 |
|-------|------|------|
| 🔴 **Critical** | 보안 취약점, 핵심 기능 누락, 구현 불가 | 즉시 수정 필수 |
| 🟡 **Major** | 품질 저하, 재작업 유발 가능 | 구현 전 수정 권장 |

## Auto-Trigger

다음 상황에서 자동으로 digging 사용을 제안:

1. `/prd` 명령 완료 직후
2. PRD 파일이 수정되었을 때
3. `/implement` 시작 전 PRD 검증 요청 시

```
💡 PRD 작성이 완료되었습니다.
   구현 전 `digging` 스킬로 계획을 검토하시겠습니까?

   예: "PRD 검토해줘" 또는 "digging"
```
