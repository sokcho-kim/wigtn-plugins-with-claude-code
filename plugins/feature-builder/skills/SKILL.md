# Feature Builder

사용자의 모호한 기능 요청을 체계적인 질문을 통해 명확한 스펙으로 변환합니다.

## When to Use

- 사용자가 새 기능을 요청할 때
- 요구사항이 모호하거나 불명확할 때
- 구현 전 스펙 정의가 필요할 때
- "로그인 기능", "검색 기능", "댓글 기능" 등 기능 관련 요청

## When NOT to Use

- 이미 명확한 스펙이 있는 경우
- 단순 버그 수정 요청
- 코드 리팩토링 요청

## Protocol

### Step 1: 프로젝트 상태 파악

다음 파일/폴더 존재 여부를 확인합니다:

**프로젝트 타입**: `package.json`, `tsconfig.json`, `next.config.*`, `nest-cli.json`
**코드 구조**: `src/` 디렉토리 구조

**상태 리포트:**

```
┌─────────────────────────────────────────────┐
│ 📊 Project Status                           │
├─────────────────────────────────────────────┤
│ Type      : [Frontend/Backend/Fullstack]    │
│ Framework : [Next.js/NestJS/Express/...]    │
│ Structure : [src 구조 요약]                 │
└─────────────────────────────────────────────┘
```

### Step 2: 핵심 의도 파악 (5W1H)

빠른 경로: 요청이 명확하면 일부 질문 생략

| 질문      | 핵심 포인트                       |
| --------- | --------------------------------- |
| **WHAT**  | 어떤 데이터/동작? 검색/CRUD/인증? |
| **WHY**   | 어떤 문제 해결? 기존 불편함?      |
| **WHO**   | 일반 사용자/관리자? 권한 필요?    |
| **WHERE** | 어느 페이지/모듈? 새로 생성?      |
| **WHEN**  | 트리거? 버튼/페이지로드/이벤트?   |
| **HOW**   | 입력→처리→출력 흐름?              |

### Step 3: 맥락별 심층 질문

**프론트엔드:**

- 반응형/다크모드 필요?
- 로딩/에러/빈 상태 처리?
- 애니메이션/접근성?

**백엔드:**

- REST/GraphQL?
- 인증 필요? 권한 체크?
- 캐싱/페이지네이션?

**풀스택:**

- 실시간 동기화?
- 낙관적 업데이트?

### Step 4: 엣지 케이스 확인

```
• 빈 값/잘못된 형식 입력 시?
• 네트워크 끊김/느린 응답?
• 동시 수정/중복 요청?
• 데이터 없음/대용량?
```

### Step 5: 스펙 정리 및 확인

```
┌─────────────────────────────────────────────────────┐
│ 📋 Feature Specification                            │
├─────────────────────────────────────────────────────┤
│ 기능명: [기능 이름]                                  │
│ 목적  : [WHY - 필요한 이유]                          │
│ 대상  : [WHO - 사용자]                               │
├─────────────────────────────────────────────────────┤
│ 📍 위치                                              │
│ • 프론트: [페이지/컴포넌트]                          │
│ • 백엔드: [모듈/엔드포인트]                          │
├─────────────────────────────────────────────────────┤
│ 🔄 동작 흐름                                         │
│ 1. [트리거] → 2. [처리] → 3. [결과]                  │
├─────────────────────────────────────────────────────┤
│ ⚠️ 엣지 케이스: [처리 방법]                          │
└─────────────────────────────────────────────────────┘

[수정] - 내용 수정
[추가] - 빠진 내용 추가
[확인] - 맞습니다, 구현 시작
```

### Step 6: 구현 계획 및 실행

```
┌─────────────────────────────────────────────────────┐
│ 📝 Implementation Plan                              │
├─────────────────────────────────────────────────────┤
│ Step 1: [작업] - [파일]                              │
│ Step 2: [작업] - [파일]                              │
│ ...                                                 │
└─────────────────────────────────────────────────────┘

[시작] - 구현 시작
[수정] - 계획 수정
```

## 기능 템플릿

### 인증 기능

```
1. 인증 방식: 이메일/비밀번호, 소셜 로그인, 매직 링크?
2. 추가 기능: 회원가입, 비밀번호 찾기, 2FA?
3. 세션 관리: JWT, 세션 쿠키, 멀티 디바이스?
```

### CRUD 기능

```
1. 기본 작업: 목록/상세/생성/수정/삭제?
2. 추가 기능: 검색/필터/정렬/페이지네이션?
3. 권한: 작성자만 수정? 관리자 분리?
```

### 검색 기능

```
1. 검색 방식: 키워드/자동완성/필터/전문검색?
2. 검색 대상: 단일 리소스/통합 검색?
3. UI: 검색바+목록/실시간 드롭다운/전용 페이지?
```

### 파일 업로드

```
1. 파일 유형: 이미지/문서/동영상?
2. 업로드 방식: 단일/다중/드래그앤드롭?
3. 저장: 로컬/S3/CDN?
4. 추가: 리사이징/썸네일/진행률?
```

## Rules

1. **점진적 질문**: 한 번에 다 묻지 않고 단계별 진행
2. **예시 제공**: 추상적 질문 대신 구체적 옵션 제시
3. **기본값 제안**: 합리적인 기본값으로 질문 최소화
4. **빠른 경로**: 명확한 요청은 질문 건너뛰기
5. **확인 필수**: 구현 전 사용자 확인 받기

## Examples

### 예시 1: 모호한 요청

```
User: "로그인 기능 추가해줘"

Claude:
1. 프로젝트 상태 확인
2. 5W1H 질문 진행
3. 인증 템플릿으로 상세 확인
4. 스펙 정리 후 확인
5. 구현 계획 제시
6. 승인 후 구현
```

### 예시 2: 명확한 요청

```
User: "이메일/비밀번호로 로그인하고 JWT 발급, /auth/login API"

Claude:
1. 프로젝트 상태 확인
2. 대부분 질문 생략 (이미 명확)
3. 엣지케이스만 확인
4. 바로 구현 계획 제시
```
